# Security Policy

## Status

**LegacyOps Console is experimental software. It is NOT production-ready
and has NOT undergone a security audit, threat-modeling exercise, or
penetration test.**

Treat this repository as a public scaffold for demos, training,
architecture validation and pilot preparation. Do NOT deploy it against
real customer data, real Siebel-like endpoints, or real billing systems
without first closing the gaps listed in
[`docs/ENTERPRISE_READINESS_GAP.md`](docs/ENTERPRISE_READINESS_GAP.md).

---

## Reporting a vulnerability

If you discover a security issue in this repository:

1. **Do NOT open a public GitHub issue.**
2. Email the maintainer privately (see the GitHub profile linked from the
   repository) with a description, reproduction steps, and impact
   assessment.
3. Allow up to 7 days for an initial response.
4. Once acknowledged, coordinate a disclosure timeline with the
   maintainer.

Please report only real security issues. For bugs, feature requests, or
documentation gaps, use the normal GitHub issue templates.

## Secrets policy

- **Never commit secrets.** This includes API tokens, passwords,
  private keys, connection strings, OAuth client secrets, and GitHub
  PATs.
- The repository ships `.env.example` with inert placeholders only.
  `.env` is in `.gitignore` and must never be committed.
- If a secret is accidentally committed, rotate it immediately and open
  a security report (see above). Force-pushing to "hide" the secret is
  NOT acceptable — assume it has already been cloned.
- Application logs must never include secrets, tokens, or full
  credentials. The current Fastify logger is configured at info level;
  audit events do not include secret material.
- Before every release, the CI workflow and the maintainer verify that
  no `github_pat_`, `ghp_`, `AKIA`, `xoxb-`, or similar known token
  prefixes appear in the diff.

### How to provide secrets locally

Copy `.env.example` to `.env`, fill in the values, and keep `.env`
outside version control. The scaffold does not require any real secrets
to run — the Fake Siebel Lab uses synthetic credentials only.

## Data policy

- **Only synthetic data.** The scaffold uses fictional customers,
  accounts, invoices, cases and Siebel-like records generated by
  `packages/demo-data`.
- **No PII.** Names, emails, phone numbers, and document numbers are
  randomly generated and do not correspond to real people.
- **No real customer data.** Do not import real customer data into this
  scaffold. The data layer is in-memory and resets on restart; it is
  not designed to protect real PII.
- **No ex-employer data.** Do not reproduce schemas, sample data, or
  internal artifacts from any current or former employer. The Siebel-like
  types in `packages/siebel-bridge/src/contracts/types.ts` are conceptual
  and original; no proprietary schema is reproduced.

## Authentication & RBAC

- **No real authentication.** The API does not implement SSO, OIDC,
  SAML, or any password-based login.
- **Simulated RBAC.** The API reads a role from the
  `x-legacyops-role` HTTP header. If absent, it defaults to `operator`.
  This is documented in `README.md` → *API RBAC Simulation* and in
  `docs/SECURITY_NOTES.md`.
- **The `x-legacyops-role` header is a demo primitive.** Any HTTP
  client can set it. It MUST be replaced by a real identity provider
  before any production-like deployment.
- The permission matrix is defined in
  `packages/permissions/src/index.ts` and enforced at the route level
  through `apps/api/src/rbac.ts`. Permission-denied events are recorded
  in the in-memory audit log.
- Real SSO/OIDC/SAML integration is a tracked enterprise gap.

---

## Integration policy

- The scaffold ships **mock adapters only**. The Fake Siebel Lab in
  `packages/siebel-bridge/src/mock/` simulates latency, session
  expiry, permission errors and conflicts without contacting any real
  backend.
- **No real Siebel REST endpoints are hardcoded.** Every connection
  parameter lives in `.env` and is inert by default.
- A real adapter implementation MUST:
  - read all connection parameters from environment variables,
  - use a service account with the minimum required privileges,
  - enforce timeouts and circuit breakers,
  - log adapter calls to the audit log,
  - never expose backend credentials to the browser.

---

## AI policy

LegacyOps Console may eventually include optional AI assistance
(currently a tracked gap, not implemented). When AI features are added:

- AI must never perform irreversible actions without explicit operator
  confirmation.
- AI output must be clearly labeled and auditable.
- AI must not bypass the RBAC permission matrix.
- Customer data must NOT be sent to external AI services without an
  explicit, reviewed data-sharing agreement and a documented legal
  basis.

```bash
pnpm audit --prod
```

This is NOT currently wired into CI. It is a tracked gap.
